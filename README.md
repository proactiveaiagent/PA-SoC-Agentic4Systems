# PA-SoC：主动型智能体加速芯片
> 基于 [Agentic4Systems GPGPU 竞赛 2026](https://github.com/ephonic/Agentic4SystemSummerSchoolContest) 的 AEC ISA
> 
> **仓库：** https://github.com/proactiveaiagent/PA-SoC-Agentic4Systems

当前的智能体（AI Agent）多属被动响应型，需用户明确指令后方可行动。基于多模态大模型与具身智能的主动型智能体能够在无需用户明确请求的情况下主动识别和推断用户的潜在需求，并即时调用工具，提供相应的援助来满足用户的需求。相比之下，主动型智能体具备更高的便捷性、效率与智能化水平，在日常生活、康养养老、居家护理、生产办公等领域具有广泛应用前景。此项目实现一个专门为主动型智能体优化的芯片，支持主动型智能体更好地完成以下操作：
1. 通过摄像头等传感器实时捕捉用户的多模态数据，给多模态大模型进行实时分析识别
2. 分析识别结果给推理大模型进行建模，预测判断用户的潜在需求，然后调用相应的工具服务满足用户的需求，给用户提供援助
3. 构建用户画像，存储在memory里，并且通过分析识别用户的所作所为更新用户画像，用于更好地进行需求识别判断和满足
4. 观察用户得到所提供支持援助后的反应，判断提供的支持援助是否正确，正确通过强化学习进行强化，不正确自我修改完善



## 一、设计目标与核心挑战

| 能力 | 硬件挑战 |
|------|----------|
| 实时多模态感知 | 摄像头/麦克风/IMU 等数据流高带宽、低延迟融合 |
| 双模型协同推理 | 感知模型（快）+ 推理模型（深）需异构并行 |
| 用户画像持久化 | 长期记忆需高容量、低延迟、可增量更新 |
| 工具调用编排 | 多服务并发调度、低延迟 I/O |
| 强化学习闭环 | 反馈信号需硬件级快速回传与权重更新 |

## 三、五大核心子系统详解

### 1. 感知前端（Perception Frontend）

**职责**：7×24 低功耗采集用户多模态数据，做硬件级预处理。

```
传感器接口
├── ISP（图像信号处理）× 2~4 路摄像头
├── AFE（音频前端）+ 波束成形 + VAD
├── IMU / 环境光 / 接近 / 心率 等融合
└── 事件触发器（Event Trigger Unit）
```

**关键设计**：

- **Always-On 感知岛（AOP Island）**：独立低功耗域，功耗 < 50mW，持续运行 VAD、运动检测、人脸/姿态粗检，仅在「有意义事件」时唤醒主计算域。
- **硬件时间对齐器（Temporal Aligner）**：将视频帧、音频块、传感器读数对齐到统一时间戳，误差 < 1ms，供多模态融合使用。
- **隐私硬件门（Privacy Gate）**：敏感区域（如密码输入、私密空间）可在硬件层做像素/音频掩码，再进入后续流水线。

### 2. 多模态分析引擎（MM-Engine）

**职责**：对感知数据进行实时多模态理解与识别。

| 模块 | 规格 | 功能 |
|------|------|------|
| **VPU（视觉处理单元）** | 128 TOPS INT8 | 目标检测、姿态、表情、场景理解 |
| **APU（音频处理单元）** | 32 TOPS | 语音识别、情感语调、环境音分类 |
| **MFU（多模态融合单元）** | 专用 Cross-Attention 硬件 | 视听对齐、跨模态语义绑定 |
| **SLM 加速器** | 支持 1~7B 参数 | 快速场景描述、意图初筛 |

**流水线设计**：

```
摄像头 30fps ──▶ VPU (5ms/帧) ──┐
                                  ├──▶ MFU ──▶ SLM ──▶ 结构化感知输出
麦克风 16kHz  ──▶ APU (10ms块) ──┘
                                    (JSON: 用户状态、环境、行为事件)
```

**输出示例**：

```json
{
  "user_state": {"emotion": "frustrated", "attention": "screen"},
  "behavior": "repeated_typing_errors",
  "context": "coding_in_ide",
  "confidence": 0.87,
  "timestamp": 1718342400.123
}
```

### 3. 推理决策引擎（Reasoning Engine）

**职责**：基于感知结果建模用户状态，预测潜在需求，生成行动计划。

```
┌─────────────────────────────────────────────────┐
│              Reasoning Engine                    │
├─────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────┐ │
│  │ 世界模型     │  │ 需求预测器   │  │ 规划器   │ │
│  │ World Model │─▶│ Need        │─▶│ Planner │ │
│  │  (7~13B)    │  │ Predictor   │  │         │ │
│  └─────────────┘  └─────────────┘  └────┬────┘ │
│                                           │      │
│  ┌─────────────┐  ┌─────────────┐        ▼      │
│  │ 用户画像     │  │ 上下文窗口   │  ┌─────────┐ │
│  │ 查询接口     │◀─│ Context Mgr │─▶│ Action  │ │
│  └─────────────┘  └─────────────┘  │ Queue   │ │
│                                     └─────────┘ │
└─────────────────────────────────────────────────┘
```

**关键硬件特性**：

- **双轨推理架构**：
  - **快轨（Fast Track）**：SLM 做毫秒级初判（「用户可能需要帮助」）
  - **深轨（Deep Track）**：LLM 做秒级深度推理（「用户卡在 API 文档，建议打开示例代码」）
- **KV-Cache 专用 SRAM**：512MB~2GB 高速缓存，减少 LLM 重复推理延迟。
- **投机解码硬件（Speculative Decode Unit）**：小模型起草 + 大模型验证，推理吞吐提升 2~3 倍。
- **需求预测专用单元（Intent Foresight Unit）**：基于历史序列的 Transformer 硬件，专门做「下一步用户可能做什么」的时序预测。

### 4. 记忆子系统（Memory Subsystem）

**职责**：构建、存储、增量更新用户画像，支撑个性化需求识别。

```
┌────────────────────────────────────────────────────────┐
│                   Memory Subsystem                      │
├────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │  工作记忆     │  │  情景记忆     │  │  语义记忆     │ │
│  │ Working Mem  │  │ Episodic Mem │  │ Semantic Mem │ │
│  │  (SRAM 64MB) │  │ (LPDDR5 4GB) │  │ (NAND 64GB)  │ │
│  │  当前会话上下文│  │ 交互历史事件  │  │ 用户画像/偏好 │ │
│  └──────────────┘  └──────────────┘  └──────────────┘ │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │  程序记忆     │  │  向量检索引擎  │  │  画像更新器   │ │
│  │ Procedural   │  │ Vector Search│  │ Profile      │ │
│  │  (工具使用模式)│  │ Engine (HNSW)│  │ Updater      │ │
│  └──────────────┘  └──────────────┘  └──────────────┘ │
└────────────────────────────────────────────────────────┘
```

**用户画像数据结构**（硬件原生支持）：

```json
{
  "user_id": "hash_xxx",
  "preferences": {
    "communication_style": "concise",
    "help_threshold": 0.6,
    "active_hours": [9, 22]
  },
  "behavior_patterns": [
    {"trigger": "typing_pause_>30s", "likely_need": "stuck_on_problem"},
    {"trigger": "repeated_scroll", "likely_need": "searching_info"}
  ],
  "skill_profile": {"coding": 0.8, "design": 0.3},
  "feedback_history": [
    {"action": "suggest_code", "outcome": "positive", "reward": +1.0}
  ],
  "embedding": [0.12, -0.34, "..."]
}
```

**关键设计**：

- **增量更新引擎（Incremental Update Engine）**：硬件加速画像向量 EMA 更新，无需全量重算。
- **遗忘曲线硬件（Forgetting Curve HW）**：按时间衰减不重要记忆，保留高价值模式。
- **隐私分片存储**：画像按敏感级别分片，支持硬件加密（AES-256）与本地联邦学习。

### 5. 工具编排引擎 + 反馈学习引擎

#### 工具编排引擎（Tool Orchestrator）

```
┌─────────────────────────────────────────┐
│         Tool Orchestrator               │
├─────────────────────────────────────────┤
│  ┌─────────┐  ┌─────────┐  ┌────────┐ │
│  │ Tool    │  │ API     │  │ Action │ │
│  │ Registry│  │ Gateway │  │ Executor│ │
│  └─────────┘  └─────────┘  └────────┘ │
│                                         │
│  支持并发调用：                          │
│  • 本地工具（打开应用、调节亮度）         │
│  • 云端 API（搜索、翻译、代码生成）       │
│  • 设备控制（IoT、智能家居）             │
└─────────────────────────────────────────┘
```

- **Tool Call 硬件解析器**：将 LLM 输出的 function call JSON 直接硬件解析为执行指令，绕过 CPU 序列化开销。
- **并行执行单元**：最多 8 路工具并发，带超时与回滚。
- **安全沙箱**：工具权限按用户画像中的信任等级动态调整。

#### 反馈学习引擎（RL Engine）

**职责**：观察用户对援助的反应，判断正确性，驱动强化或自我修正。

```
用户收到援助后的反应
        │
        ▼
┌───────────────────┐
│  反应观察器        │  ← 复用 MM-Engine 持续感知
│  Reaction Observer │
└────────┬──────────┘
         │
         ▼
┌───────────────────┐     正确      ┌──────────────────┐
│  奖励评估器        │─────────────▶│  策略强化          │
│  Reward Evaluator  │              │  Policy Reinforce  │
│                   │              │  (LoRA 权重更新)   │
│  信号：            │              └──────────────────┘
│  • 表情/语调变化   │
│  • 行为改变        │     不正确    ┌──────────────────┐
│  • 显式反馈        │─────────────▶│  自我修正          │
│  • 任务完成率      │              │  Self-Correction   │
└───────────────────┘              │  (反思 + 策略调整)  │
                                  └──────────────────┘
```

**奖励信号硬件化**：

| 信号类型 | 检测方式 | 奖励值 |
|----------|----------|--------|
| 用户微笑/放松 | 表情识别 | +0.8 |
| 继续使用/采纳建议 | 行为追踪 | +1.0 |
| 显式「谢谢」 | 语音识别 | +1.0 |
| 关闭/忽略援助 | 行为追踪 | -0.5 |
| 显式拒绝 | 语音/手势 | -1.0 |
| 重复相同困境 | 模式匹配 | -0.7 |

**硬件加速 RL**：

- **在线 LoRA 更新单元**：无需全量微调，仅更新低秩适配器（< 1% 参数量），毫秒级完成。
- **经验回放缓冲区（Replay Buffer）**：片上 256MB SRAM，存储最近 10K 条 (state, action, reward) 元组。
- **反思生成器（Reflection Generator）**：失败时触发 LLM 生成「为什么错了」的反思文本，写入程序记忆，避免重复犯错。

## 四、完整数据流（端到端时序）

```
时间轴 ──────────────────────────────────────────────────────────▶

T+0ms    传感器捕获用户行为（反复删改代码）
         │
T+5ms    AOP 岛检测到异常行为模式，唤醒主域
         │
T+15ms   MM-Engine 输出：{behavior: "stuck", emotion: "frustrated"}
         │
T+20ms   快轨 SLM 初判：可能需要编码帮助（confidence: 0.72）
         │
T+25ms   Memory 检索：用户画像显示偏好「简洁代码示例」
         │
T+50ms   深轨 LLM 推理：预测需求 = 「API 用法示例」
         │
T+55ms   Planner 生成行动计划：调用 code_search + 展示示例
         │
T+60ms   Tool Orchestrator 并发执行
         │
T+200ms  援助呈现给用户（弹出相关代码示例）
         │
T+500ms  观察用户反应：停止删改，开始阅读示例
         │
T+800ms  Reward Evaluator：正向信号（+0.9）
         │
T+810ms  RL Engine：强化「stuck + frustrated → code_example」策略
         │
T+815ms  Profile Updater：更新用户行为模式权重
```

**端到端延迟**：从感知到首次援助 < 200ms（快轨），深度援助 < 1s。

## 五、芯片规格建议

| 参数 | 规格 |
|------|------|
| 工艺 | 3nm / 4nm |
| 主算力 | 200+ TOPS (INT8)，50+ TFLOPS (FP16) |
| 专用 NPU | 多模态 128T + 推理 72T |
| 片上 SRAM | 512MB（KV-Cache + 工作记忆 + Replay Buffer） |
| 片外内存 | LPDDR5X 16~32GB + UFS 128GB |
| 功耗 | AOP 岛 < 50mW；主动推理 < 8W；峰值 < 15W |
| 互连 | PA-Fabric 2TB/s 片上 NoC |
| 安全 | TEE + 硬件隐私门 + 加密画像存储 |

## 六、与通用 AI 芯片的核心差异

| 维度 | 通用 AI 芯片 | PA-SoC |
|------|-------------|--------|
| 工作模式 | 请求→响应 | 始终在线→主动预测 |
| 感知 | 按需启动 | Always-On 低功耗岛 |
| 模型 | 单模型推理 | 双轨快慢协同 |
| 记忆 | 无/短期 | 四层记忆 + 画像增量更新 |
| 输出 | 文本/图像 | 工具调用 + 物理世界干预 |
| 学习 | 静态权重 | 在线 RL + LoRA 自我修正 |
| 隐私 | 云端为主 | 端侧画像 + 硬件隐私门 |

## 总结

**PA-SoC** 的本质不是「更快的推理芯片」，而是把主动智能体的完整认知循环固化到硅片上：

1. **感知岛** — 永远看着用户
2. **双轨引擎** — 快判 + 深想
3. **记忆系统** — 越用越懂用户
4. **工具编排** — 从建议到行动
5. **RL 闭环** — 从错误中成长
